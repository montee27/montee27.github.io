The Birthday Paradox

Statistics show that it is over 50% likely that two people in the same room will share the same birthday given that the number of people is larger than 23. Our program creates a simulation which tracks stastically how probable it is for this paradox to occur on a sample size of n people with 20 trials each. Outputs of probabilities for a sample size of n people is given in increments of 5, all the way to 100.

This project runs a high level simulation of the statistical likelihood of the Birthday Paradox appearing in sample sizes of 5, 10, 15, .. , 100 people that have randomly generated birthdays over 20 trial runs.

This program creates random lists of n people in increments of 5 all the way to 100. It then runs an expirement with 20 trials each on each of the sample sizes, calculating and outputting the resulting statistical likelihood of the Birthday paradox occuring over 20 runs on a sample size of n.

This project uses a high-level Simulator class which creates a random object that is passed to the expirement class, and prints out ending results. The expirement class takes in the number of people in a given expirement and the number of trials, as well as the random object created in Simulator in it's constructor, and includes a method which returns a given expirements result, this method is used in a for-loop in the top-level Simulator class. The Trial class models a given trial in our expirement, it's constructor takes in the number of people in the trial as well as the random object created in Simulator. Trial has a method which uses a nested for-loop to iterate through a list of random birthdays and checks if there is a duplicate. This method is utilized in the for-loop which obtains the expirement results in the Expirement class. Finally, the birthday class simply takes in the random object that was created in Simulator in its constructor, and has another method which checks if a given month is in a certain month type (29 days, 30 days etc.), this method is used when creating random birthday objects in the constructor. Finally, the Birthday class overrides the inherited Object class's equals method, implementing it in a manner that simply checks if the values of the date and the month are the same.

This project helped me further understand the Single Responsibility Principle (SRP) as we were required to make our program modular so that debugging functionality would not be an issue.

Elevator Simulator

This project creates a simulation of multiple elevator types handling different calls in a building. There are 4 different types of Elevators and 2 different types of Schedulers. Each elevator has different behaviors as well as the schedulers. Specifically, this project is best implemented through the use of Abstract classes and interfaces which allow us to group common functionality/attributes for a robust and abstracted design. This ensures that we follow the SRP as each of our classes is doing only one thing, and having only one reason to change. It supports the open closed principle as our classes are programmed to abstractions (method taking in IElevator). Since we took time to properly design parent/subclass structure, substituting parent class objects for subclass objects wont break our program. For dependency inversion, we ensured that our high level scheduler relies on low level elevators. Finally, our interfaces were created narrowly enough so that implementing classes did not have to implement necessary methods.

This project models a simulation of different types of elevators that respond to calls which are sent by a scheduler. In standard scheduling, the scheduler always begins assigning calls to the left-most elevator and iterate through accordingly. In complex scheduling, however, the scheduler assigns the call to the elevator that can complete it in the least amount of rounds. There are four different types of elevators: 1) The standard elevator which visits every floor and will not change direction to pick up a new call 2) The Freight elevator can visit every floor. This elevator only takes calls when empty, and won't service more than one person at a time, similar to the standard elevator, it will not change directions during a call. 3) The penthouse elevator can only travel between the top and bottom floor and can only change direction if it is empty. 4) Finally, the express elevator only services the top half of the building, only accept calls if its in the same direction as it is going, and change directions to answer a call only if the elevator is empty.

Above we have described how our design makes use of the SOLID principles. As for the four programming pillars, we 1) ensured that our design was abstracted through the use of proper parent and subclass relationship/structure 2) Since we did not users to be able to access/modify attributes of the Scheduler class, we made these attributes private/protected, supporting polymorphism 3) We utilized object polymorphism as a specific elevator can be described as type Elevator (elevator Abstract class) 4) Through our use of inheritance we were properly able to abstract our design.

In this project, I was responsible for implementing the express and standard elevator, as well as the complex scheduler. While following Test-Driven Development (TDD) I faced certain issues with code coverage, specifically, I had issues with being able to properly mock the Call class dependency with my current code structure. Refactoring the code such that my dependencies were properly mocked produced much cleaner code than what I had before, and is a facet of TDD that I enjoyed. Implementing the express and standard elevator was a great form of practicing the SOLID programming principles. Specifically, refactoring the logic in my conditionals to be less wordy and call more helpers, as well as setting a few boolean values for readability, was a great way to practice producing code which is abstracted; so that any user can look at it and understand the fundamental logic.

This project gave me great insight on how a real life elevator scheduling system would work, and taught me important skills regarding the continous development of software. This was also my first experience with test-driven development, and my first opportunity to see first-hand how this process produces robust and clean code. This project also taught me that the design phase is just as important, if not more important, than the implementation phase. If we had not properly laid out our class structure prior to implementation, we could have potentially produced code which violates the SOLID programming principles.
